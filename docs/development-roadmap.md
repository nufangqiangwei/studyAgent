# 开发路线

路线按学习阶段组织。每一阶段都应该可以运行、可以测试、可以总结。

## Phase 0: 项目初始化

目标：建立项目骨架和文档。

- 创建 `AGENTS.md` 和 `docs/` 文档。
- 明确产品方向和模块边界。
- 保持 `main.go` 简单可运行。
- 创建第一版代码骨架。

完成标准：

- 能通过 `go run .` 运行当前程序。
- 文档能说明项目目标、架构和下一步。
- 启动参数、命令注册、agent loop、prompt、LLM、日志模块已解耦。

当前状态：已完成。

## Phase 1: CLI 骨架

目标：实现基本命令行入口。

- 支持 `agent run "任务描述"` 形式。
- 支持无参数进入交互模式。
- 抽象输入输出接口，便于测试。
- 将入口迁移到 `cmd/agent/main.go`。

完成标准：

- 用户可以在终端输入任务。
- CLI 层不包含 agent 核心逻辑。
- 新命令可以通过 command registry 注册。

当前状态：已完成命令模式，交互模式尚未实现。

## Phase 2: 工具系统

目标：实现最小工具注册和调用。

- 定义 `Tool` 接口。
- 实现工具注册表。
- 实现只读文件工具。
- 实现文本搜索工具。
- 给每次工具调用记录输入、输出和错误。

完成标准：

- agent core 可以通过统一接口调用不同工具。
- 工具可以独立测试。

## Phase 3: LLM 接口

目标：接入模型调用，但不绑定具体供应商。

- 定义 `llm.Client` 接口。
- 定义请求和响应结构。
- 实现一个 mock client 用于测试。
- 再实现真实供应商 client。
- 维护 OpenAI、DeepSeek、Gemini、Claude/Anthropic 独立请求构建模块。

完成标准：

- 单元测试不需要真实 API key。
- 更换模型供应商不影响 agent core。

当前状态：已完成统一接口、mock client、provider 请求构建模块和非 mock provider 的真实 HTTP 调用。

## Phase 4: Agent 循环

目标：实现最小可用 agent。

- 接收用户任务。
- 构建模型请求。
- 解析模型响应。
- 根据响应调用工具。
- 把工具结果回传模型。
- 输出最终回答。

完成标准：

- 可以完成简单的项目问答任务。
- 可以读取文件并基于内容回答。

## Phase 5: 代码修改能力

目标：让 agent 能提出并应用代码修改。

- 设计补丁或文件写入工具。
- 修改前读取文件。
- 修改后运行格式化。
- 支持运行测试命令。
- 将测试失败反馈给 agent。

完成标准：

- 可以完成小型代码修改任务。
- 可以报告修改文件和验证结果。

## Phase 6: 会话和配置

目标：增强真实使用体验。

- 支持多轮会话。
- 支持配置文件。
- 支持模型、超时、工作区路径等参数。
- 支持会话历史裁剪。

完成标准：

- 用户可以连续追问。
- 配置不需要硬编码。

## Phase 7: 进阶能力

目标：探索更接近成熟 CLI agent 的能力。

- 流式输出。
- 权限确认。
- 工具调用审计。
- 插件化工具。
- 任务计划展示。
- 更细粒度的上下文管理。
- 长任务恢复。

完成标准：

- 项目形成可持续扩展的 agent CLI 原型。
- 每个进阶能力都能独立开关和测试。
